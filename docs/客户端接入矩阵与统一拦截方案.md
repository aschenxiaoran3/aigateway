## 统一拦截分层

### 第一层：直连网关层

适用于支持自定义 OpenAI-compatible Base URL 的客户端：

- OpenClaw
- Hermes
- 后续内部 IDE 插件、测试工具、Agent 工具

统一出口：

- `http://127.0.0.1:3001/v1`

统一要求：

- 强制带 `X-AI-Gateway-Client`
- 强制带 `metadata.purpose`
- 推荐带 `trace_id / project_code / artifact_type`

### 第二层：系统代理层

适用于不方便直接改模型出口的客户端：

- Cursor
- Codex
- 未来公司内部其他桌面 AI 客户端

本仓新增：

- `gateway-egress-proxy`

职责：

- 接收系统 `HTTP_PROXY / HTTPS_PROXY`
- 记录所有已知 LLM 域名访问
- 支持两种模式：
  - `observe`
  - `enforce`
- 在 `enforce` 模式下阻断直连外部 LLM 域名

注意：

- 当前实现是“域名级拦截与审计”，不是 HTTPS 解密
- 若要审查具体 prompt / response，仍需：
  - 客户端直连公司网关
  - 或企业级 TLS 代理 / 受信 CA / MITM 方案

### 第三层：企业出口控制层

部署到公司内网后，真正的策略应该是：

1. 内网客户端只允许访问公司网关
2. 直连 `api.openai.com / chatgpt.com / api.anthropic.com / dashscope.aliyuncs.com / openrouter.ai` 等域名全部阻断
3. 门禁、审计、配额、成本、项目挂接全部在公司网关内完成

## 当前实现物

1. `ai-gateway`
   - 统一入口
   - `trace_id / pipeline_run_id / project_code` 留痕
   - 门禁结果同步 control-plane

2. `control-plane`
   - 项目治理
   - 控制平面
   - 运行编排
   - 证据包
   - 阶段验收

3. `gateway-egress-proxy`
   - 统一代理入口
   - LLM 域名识别
   - 代理日志入库 `gateway_egress_events`

4. 审计脚本
   - `scripts/audit-client-interception.js`
   - 用于审计本机客户端是否接入网关

5. PAC 生成脚本
   - `scripts/generate-llm-proxy-pac.js`
   - 用于给系统代理或企业浏览器下发 PAC 文件
