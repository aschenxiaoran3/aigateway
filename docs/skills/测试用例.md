## 1. 测试策略与总体计划

### 1.1 测试类型与阶段

| 阶段 | 说明 | 与本方案用例的对应关系 |
|:-----|:-----|:------------------------|
| **单元测试** | 开发完成单测，覆盖率目标 ≥ 90% | 与第 19 章「单元测试与测试用例对齐说明」对齐；由开发在代码层执行 |
| **集成测试** | 验证模块间交互（如采购入库后库存、批次、流水联动） | 采购入库（TC-001～TC-003）、调拨（TC-105、TC-106）、销售出库（TC-202～TC-204）、批次（TC-301、TC-304、TC-305）等跨表/跨服务场景 |
| **系统测试** | 验证完整业务流程与业务规则 | 本方案第 4～16 章主体用例 |
| **回归测试** | 版本发布前回归 | 范围：所有 P0 用例 + 上次迭代缺陷关联用例 + 核心接口（见 1.4 自动化建议） |
| **性能/稳定性测试** | 响应时间、并发、长时间运行 | 第 15 章、第 17.1 节 |

### 1.2 测试入口标准与出口标准

**入口标准（满足后方可启动系统测试）**：

- 测试环境部署完成，lime-bill-service 可正常访问
- 核心接口可通（如采购入库单创建/审核、调拨单创建/审核出库与入库、库存查询）
- 基础测试数据就绪（见 2.4 测试数据字典与基线数据）
- 相关文档齐全：PRD、技术方案、前端 API 接口文档、数据库设计

**出口标准（满足后方可发布）**：

- 所有 P0 用例 100% 通过
- 无致命（Critical）、严重（Major）级别缺陷
- 核心业务流程测试覆盖率 100%（采购入库、调拨、销售出库、批次、库存查询）
- 性能指标满足 PRD/技术方案要求（如库存查询 < 500ms，更新操作 < 1s）
- 数据一致性用例（第 16 章、第 18.1 节）通过

### 1.3 测试环境与数据策略

- **环境依赖**：本地/测试环境 MySQL 8.0+；若对接供应商系统、WMS 等第三方，需明确 Mock 或联调策略
- **数据准备**：使用基线数据脚本或数据工厂（如 `lime-bill-service/sql/init.sql`、项目内测试 SQL）准备标准商品、仓库、批次；见 2.4 测试数据字典
- **数据隔离**：建议使用独立测试库或通过测试专用租户/前缀隔离数据，避免与开发/生产混用及并行测试相互干扰

### 1.4 自动化测试策略

以下高频、核心、稳定的用例建议优先接入 CI/CD 自动化回归：

- **采购入库**：TC-001、TC-002、TC-003（创建→提交→审核通过，验证库存与流水）
- **调拨**：TC-105、TC-106（审核出库、审核入库，验证源仓/目标仓库存与流水）
- **销售出库**：TC-202（FIFO 扣减单批次）
- **批次**：TC-301（创建批次发布即生效）、TC-301-1（创建批次-可选入库单含调拨单，接口与数据范围）、TC-304（批次作废返还库存）
- **数据一致性**：TC-1101、TC-1102、TC-1103（流水完整性、商品与批次数量一致）

自动化实现方式可按项目选型（如 JUnit + 接口调用、Postman/Newman、JMeter 等）。

|:-----|:----|

### TC-1107：库存汇总与明细恒等式及展示准据（AC-4.5.4/4.5.5）

**测试目的**：验证库存明细接口返回的汇总与明细满足数量恒等式；若约定以明细合计覆盖汇总，则 summary 的 allocatable/allocated 与明细合计一致。

- **展示准据**：若实现约定「以明细合计覆盖汇总」，则 summary.allocatableStock = Σ(distributions[].allocatableStock)，summary.allocatedStock = Σ(distributions[].allocatedStock)；summary.totalStock 仍可与库存表一致（与流水一致）。
- 明细列表中每行 totalStock = allocatableStock + allocatedStock（按行成立）。

| **基准性能** | 单接口响应时间 | 库存查询 < 500ms（PRD 要求）；复杂列表（如批次分布）< 2s |
| **压力测试** | 大促场景 | 高频并发创建销售出库单，验证库存扣减正确性、系统吞吐量、无超卖 |
| **稳定性测试** | 长时间运行 | 混合执行采购、调拨、销售、批次操作，监控内存、事务堆积、错误率 |

与现有用例的对应：第 15 章 TC-1001（库存查询性能）、TC-1002（并发更新性能）可作为基准；扩展时可采用 JMeter/Gatling 等工具构造负载。

### 17.2 安全与权限测试

| 场景 | 验证点 |
|:-----|:-------|
| **接口权限** | 不同角色（仓管员、财务、管理员）对采购单、调拨单、批次的增删改查权限是否符合设计 |
| **数据权限** | 若系统按仓库/租户隔离，验证用户仅能操作其所属范围数据 |
| **敏感数据** | 成本等敏感字段在前端/接口响应中的脱敏或权限控制 |

建议在系统测试中补充角色矩阵与权限用例，或单独编写权限测试用例集。

### 17.3 兼容性与部署验证

- **数据库**：若支持多数据库（如 MySQL/PostgreSQL），需验证 SQL 与建表脚本兼容性
- **部署**：新版本上线后，验证旧版本数据的迁移脚本或兼容性（如有表结构变更）

## 18. 库存专项测试（事务一致性、幂等性、边界与监控）

### 18.1 事务一致性与幂等性

| 场景 | 验证目的 | 建议步骤 |
|:-----|:---------|:---------|
| **分布式/局部失败** | 数据最终一致性 | 在审核入库（TC-003）或调拨出库（TC-105）时，模拟库存表更新成功但流水表写入失败（或通过 Mock 延迟/失败），验证是否有补偿或回滚，数据是否一致 |
| **幂等性** | 防重复扣减 | 同一审核出库请求（相同单据、版本）多次提交，验证库存仅扣减一次；对消息重试、前端重复提交场景至关重要 |

可与开发确认是否已有事务与幂等设计，并据此在第 18 章或用例库中补充具体用例步骤与预期。

### 18.2 边界与极限值测试

| 场景 | 说明 |
|:-----|:-----|
| **库存为 0** | 销售出库、调拨出库时库存为 0，应拦截并提示库存不足 |
| **库存极大值** | 如 9999999，验证运算与展示无溢出或性能问题 |
| **批次有效期** | 商品恰好在到期日当天进行出库或调拨时的业务逻辑与提示 |

### 18.3 监控与可观测性验证

- 在性能或异常测试中，验证关键业务指标（如库存变化量、订单处理成功率）是否按约定上报监控（如 Prometheus/Grafana）
- 验证错误日志是否包含足够定位信息（单据号、用户 ID、操作参数等），便于线上排查

## 19. 单元测试与测试用例对齐说明

本节说明《库存管理系统测试方案》与 lime-bill-service 单元测试（Spock）的对应关系，便于回归时确认：**测试用例中的场景是否均有单测覆盖**，以及**单测场景是否在测试用例中有对应条目**。PRD 验收标准（AC-4.x.x）与单测的对应见 **3.1 PRD 验收标准（AC）与测试用例对照表**；单测中应对「关联关系与一致性」「数量恒等式」做显式断言。

### 19.0 PRD 验收标准（AC）与单测断言建议

单测中建议对以下内容做**可重复的断言**，与 PRD 验收标准一致，防止后续改动破坏约定：

| PRD 验收类型 | 单测断言建议 | 已覆盖的 Spec 示例 |
|:-------------|:-------------|:--------------------|
| **数量恒等式** total_stock = allocatable_stock + allocated_stock | 产出汇总 DTO 的 Convert/Service/Biz 单测中，对满足恒等式的输入做结果恒等式断言 | InventoryBizQueryServiceImplSpec：getInventoryDistribution 汇总的可分配与已分配以明细合计为准、无入库明细时 summary 满足恒等式；InventoryBatchQueryConvertSpec：toSummaryDto 汇总结果满足恒等式 |
| **汇总与明细一致性**（以明细合计覆盖汇总） | 当接口同时返回 summary 与 distributions 时，断言 summary.allocatableStock = Σ(distributions.allocatableStock)、summary.allocatedStock = Σ(distributions.allocatedStock)（若约定以明细为准） | InventoryBizQueryServiceImplSpec：getInventoryDistribution 汇总的可分配与已分配以明细合计为准 |
| **关联关系**（单据→流水、变动量一致） | 审核类流程单测或集成测试中，Mock 后断言调用库存/流水服务的参数与单据明细数量一致；或集成测试中查库校验 | PurchaseInBizServiceImplSpec、TransferBizServiceImplSpec 等编排类 Spec 可补充「调用 inventoryUnifiedService 时传入的 quantity 与明细合计一致」等交互断言 |
| **错误信息含 productId/warehouseId** | 库存不存在或可分配不足时，断言异常 message 包含 productId、warehouseId | InventoryUnifiedServiceImplSpec：库存不存在时抛 INVENTORY_NOT_FOUND 且 message 含 productId、warehouseId |
| **数据范围/多数据源**（同一场景多接口、返回含多类型） | 查询接口数据范围含多种类型（如可选入库单含采购+调拨）时，单测 Mock 多类型数据源，断言 result 中**同时包含**各类型（如 list 中既有 IN- 也有 TR- 的 billCode） | TransferOrderQueryServiceImplSpec：listSelectableInboundItems 可同时返回采购入库单与调拨单明细（对应 AC-4.4.10） |

新增或修改单测时，可在 feature 的 Javadoc 中注明对应 PRD AC 编号（如 `对应 PRD AC-4.5.4、AC-4.5.5`），便于追溯。

- **建议补充单测**：TC-302 查询批次详情（InventoryBatchBizQueryServiceImpl / getDetail 或 QueryService.getDetail）；TC-002 提交采购入库单审核（若后端有提交审核接口，可补 Biz/Service 单测）。

- **已对齐**：批次创建/作废/调整、批次流水查询、采购入库创建/审核/查询、调拨创建/提交校验/审核出库入库、异常（批次状态、调拨状态、同一商品多批次）等，单测与测试用例已对应；单测中「现有明细调节 + 新纳入」「净减少」「delist 异常不记日志」等与测试用例描述一致。
- **PRD 验收标准同步**：关联关系与一致性用例 TC-1104～TC-1107 对应 AC-4.1.6、AC-4.2.6、AC-4.3.4、AC-4.5.4/4.5.5；AC-4.4.10（创建批次-可选入库单须含调拨单）对应 TC-301-1 与 TransferOrderQueryServiceImplSpec「listSelectableInboundItems 可同时返回采购与调拨单明细」单测；新增单测时请在 Javadoc 中标注对应 AC 编号。
- **单元测试规范-数据范围与多数据源**：当某查询接口的**数据范围**包含多种类型或多种来源（如可选入库单含「采购入库单已入库 + 调拨单已完成」）时，单测须包含至少一条用例：在 Mock 多类型数据源的前提下，断言返回结果**同时包含**各类型或符合约定范围（例如 list 中既有 IN- 也有 TR- 的 billCode），避免后续重构或误改导致数据范围收窄。参见 [接口与数据展示-规避清单](../../standards/接口与数据展示-规避清单.md) 第 4 节。

## 21. 测试执行计划

### 21.1 测试阶段

1. **单元测试**：开发阶段，覆盖率 ≥ 90%
2. **集成测试**：功能开发完成后
3. **系统测试**：所有功能完成后
4. **性能测试**：系统测试通过后
5. **回归测试**：每次版本发布前

### 21.2 测试优先级

- **P0**：核心功能，必须通过
- **P1**：重要功能，建议通过
- **P2**：一般功能，可选

### 21.3 测试通过标准

- P0 用例 100% 通过
- P1 用例 95% 以上通过
- 无致命（Critical）、严重（Major）级别缺陷
- 性能指标达标（见 PRD/技术方案）

### 21.4 缺陷管理流程

- **提交规范**：标题简明、重现步骤、预期/实际结果、截图或日志、环境信息
- **严重等级**：致命（系统不可用/数据损毁）、严重（核心功能不可用）、一般（功能异常可绕行）、建议（体验与改进）
- **流转**：提交后由开发确认与修复，回归通过后关闭；严重及以上需产品/测试评审通过后方可发布

### 21.5 风险分析与应对

| 风险 | 应对措施 |
|:-----|:---------|
| 第三方接口（供应商/WMS）不稳定 | 建立 Mock 服务或契约测试，保证核心路径可测 |
| 测试数据构造复杂（多批次、多仓库） | 使用基线数据脚本与 2.4 数据字典，必要时开发数据准备工具或 Factory |
| 并发场景难模拟 | 使用 JMeter/Gatling 或脚本并发调用接口，验证乐观锁与无重复扣减 |
| 环境与生产差异 | 明确测试环境与数据隔离策略，关键场景可在预发环境复测 |
