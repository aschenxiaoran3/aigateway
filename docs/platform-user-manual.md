# AI 工程化管理平台使用手册

## 1. 这套平台是干什么的

AI 工程化管理平台用于把需求澄清、方案生成、知识沉淀、AI 执行、门禁校验、运行留痕和阶段验收收口到同一套工作台。

当前平台已经形成 3 条可协同的主链路：

- Deep Wiki：把仓库、版本线、快照、图谱和知识页面沉淀成可复用的工程知识底座。
- 文档门禁：围绕 PRD、技术方案、测试方案执行标准化门禁与产物发布。
- Harness：把“一个具体需求/Bug/收口任务”转成可流转卡片，并串起人工确认、AI Runtime、单测、UAT 与总结。

一句话理解：

`Deep Wiki 负责看懂系统，文档门禁负责产出方案，Harness 负责把单个任务收口。`

## 2. 适合谁用

- 平台管理员：初始化数据、维护规则、观察各模块是否健康。
- 产品/方案负责人：上传 PRD、生成技术方案和测试方案、推进需求确认。
- 开发负责人：在 Harness 中组织任务收口，查看 Runtime 和单测结果。
- 测试/UAT：查看待验收任务、给出通过/打回结论，并生成阶段总结。
- 项目经理：在项目治理、运行编排、阶段验收中看整体进度和风险。

## 3. 页面导航说明

进入 `http://127.0.0.1:3000/` 后，左侧主要菜单建议这样理解：

- `总览驾驶舱`：看平台总体运行状态。
- `项目治理`：维护项目、波次、版本线、风险和进展。
- `控制平面`：看标准节点、控制配置、契约和平台定义。
- `运行编排`：看 trace 级别的链路推进和阻断点。
- `门禁治理`：看规则与执行治理。
- `文档门禁`：处理 PRD、技术方案、测试方案的生成与校验。
- `Deep Wiki`：做多仓知识沉淀、快照发布和图谱浏览。
- `AI 协同研发`：也就是 Harness/Flowboard 工作台。
- `度量与可观测`：看平台用量、趋势和效率指标。
- `知识与审计`：看知识资产、入库状态、RAG 查询和抽检记录。
- `阶段验收`：看项目级 evidence pack、验收结论和提效口径。
- `日志查询`、`API Key 管理`、`团队管理`、`系统设置`：偏平台治理和运维。

```bash
npm run phase1:migrate

```

### 4.2 最小启动服务

至少需要这 4 个服务：

- `ai-gateway`
- `control-plane`
- `knowledge-base`
- `admin-ui`

### 4.3 首次登录建议检查

1. 进入 `控制平面`，确认基础配置可见。
2. 进入 `知识与审计`，确认平台手册、规则说明等资产已入库。
3. 进入 `Deep Wiki`，确认有仓库、运行记录或项目样板。

如果页面能打开但数据全空，优先检查 `control-plane` 和 `ai-gateway` 是否已启动。

## 5. Harness 是什么，怎么用起来

### 5.1 Harness 的定位

Harness 面向“单个工作项”的闭环推进，适合处理这些场景：

- 一个明确的需求收口
- 一个 Bug 修复闭环
- 一次 Deep Wiki 或文档门禁联动后的开发验收
- 一次本地仓库回归、收口或 UAT 打回后的返工

它不是项目管理大盘，也不是代码托管工具，而是卡片化的 AI 协同执行台。

### 5.2 进入入口

左侧进入 `AI 协同研发`。

页面顶部有 3 个联动入口：

- `打开 Deep Wiki`
- `文档门禁（PRD→技术→测试）`
- `按 ID 打开文档门禁`

这意味着 Harness 不是孤立页面，而是和 Deep Wiki、文档门禁串在一起用的。

### 5.3 新建卡片前先准备什么

建议至少准备下面二选一：

- `repo_url`：本地绝对路径或 git 地址
- `deepwiki_run_id`：已经存在的 Deep Wiki run

最佳实践是优先使用本地绝对路径仓库，例如：

```text
/Users/xiaoran/.openclaw/workspace/projects/ai-platform
```

这样 Harness Runtime 能直接读取当前本地分支和提交信息，不需要额外做远端仓库预取。

如果这张卡片是从 Deep Wiki 任务延伸出来的，也可以直接绑定 `deepwiki_run_id`，让 Runtime 从 Deep Wiki 快照对应的本地副本启动。

可选项：

- `bundle_id`：如果已经有文档门禁任务，可以挂到已有 bundle，方便跳转继续处理。

### 5.4 新建卡片字段说明

点击 `新建卡片` 后，主要字段含义如下：

- `标题`：这次任务的业务名称，建议写清楚目标。
- `类型`：当前支持 `需求`、`Bug`、`任务`。
- `优先级`：高、中、低优先。
- `repo_url`：本地绝对路径或 git 仓库地址。
- `repo_branch`：预期分支名；如果使用本地绝对路径，Runtime 会尽量自动识别当前分支。
- `deepwiki_run_id`：绑定已有 Deep Wiki run。
- `bundle_id`：绑定已有文档门禁任务。
- `需求摘要`：这次要做什么、为什么做、验收边界是什么。

创建成功后，卡片默认进入 `demand_confirm_wait`，也就是“等待需求确认”。

### 5.5 Harness 的标准流转

当前 V1 的标准状态大致如下：

`demand_confirm_wait -> design_confirm_wait -> development_coding -> development_unit_testing -> uat_wait -> deploy_pending`

异常和返工分支：

- `exception`：Runtime 或单测失败
- `returned_to_dev`：UAT 打回后重新返工

人工 checkpoint 共 3 个：

- `demand_confirmation`
- `design_confirmation`
- `uat_acceptance`

### 5.6 页面上怎么操作

#### 第一步：确认需求

在卡片详情的 `操作面板` 中输入补充说明，然后点击 `确认需求`。

效果：

- 当前需求 checkpoint 被消化
- 卡片流转到 `design_confirm_wait`
- 系统写入一条 AI 设计建议消息

适用场景：

- 原始摘要已经足够清晰
- 你已经在线下完成了需求澄清
- 希望尽快进入设计确认与执行阶段

#### 第二步：确认设计

输入设计确认或边界说明后，点击 `确认设计`。

效果：

- 设计 checkpoint 被消化
- 卡片进入 `development_coding`
- 后台自动创建一次 Runtime run

这一按钮是 Harness 的正式启动点。对最终用户来说，可以把它理解成“允许 AI 开始执行”。

#### 第三步：观察 Runtime

卡片详情里会出现这些信息：

- `最近一次 Runtime`
- `Runtime 日志`
- `最近 Summary`

Runtime 会做几件事：

1. 解析仓库来源
2. 准备独立工作区副本
3. 记录本次 change request
4. 自动识别测试命令
5. 执行单元测试
6. 生成一份 AI Runtime 变更总结

当前支持自动识别的测试命令包括：

- `npm test`
- `./mvnw -q -DskipTests=false test`
- `mvn -q -DskipTests=false test`
- `./gradlew test`
- `gradle test`

#### 第四步：做 UAT 结论

当卡片进入 `uat_wait` 后，可以在操作面板中选择：

- `UAT 通过`
- `UAT 打回`

如果 `UAT 通过`：

- 卡片进入 `deploy_pending`
- 系统会自动生成一份最终变更总结

如果 `UAT 打回`：

- 卡片进入 `returned_to_dev`
- 后台会再次启动 Runtime，按打回意见进入下一轮返工闭环

### 5.7 手动启动 Runtime 什么时候用

按钮 `手动启动 Runtime` 适合这些场景：

- 想在不重新走设计确认的情况下补跑一次
- UAT 已经给了口头反馈，想先快速验证
- 要针对同一张卡补充一轮回归检查

一般做法是：

1. 在操作面板输入本轮变更说明
2. 点击 `手动启动 Runtime`
3. 到 `Runtime 日志` 里观察执行结果

### 5.8 当前 Harness V1 的真实边界

这部分非常重要。

当前 UI 已经能完成“卡片流转 + Runtime 工作区准备 + 测试执行 + UAT 收口 + 总结生成”，但它还不是一个通用自动改码 agent。

当前行为是：

- 如果只通过页面输入 `change_request`，Runtime 默认会在工作区写入 `.ai-harness/change-request.md`
- 然后继续识别测试命令并执行测试
- 最终生成运行总结

也就是说，默认模式更偏“记录变更意图并跑闭环”，而不是“直接改业务文件”。

只有后端调用 `POST /api/v1/harness/cards/:id/runtime/start` 并传入结构化 patch 参数时，才会直接改目标文件，例如：

- `target_file`
- `replace_before`
- `replace_after`
- `append_content`
- `full_content`

因此，面向业务团队培训时，建议把 Harness V1 定位为：

- 一个可落地的任务流转与验收台
- 一个可联动本地仓库和测试命令的最小 AI Runtime
- 一个为后续接入真正代码 agent 预留扩展位的控制界面

不要把它对外描述成“已经全自动修改代码并提交”的成熟能力。

### 5.9 Harness 最佳实践

- 优先绑定本地绝对路径仓库，闭环最稳定。
- `需求摘要` 尽量写清楚目标、范围、验收标准。
- 设计确认前，把边界条件和不可改动约束写进操作面板。
- UAT 打回时，备注尽量具体，便于下一轮 Runtime 使用。
- 如果这是知识驱动型任务，先在 Deep Wiki 把快照和图谱沉淀好，再建 Harness 卡片。
- 如果这是文档驱动型任务，先在文档门禁形成 PRD/技术/测试方案，再回到 Harness 做开发收口。

## 6. Deep Wiki、文档门禁、Harness 如何串起来

推荐的产品化闭环如下：

### 路径 A：知识先行

1. 在 `Deep Wiki` 中选择项目/仓库/版本线。
2. 生成或发布快照，确认图谱、页面和摘要可用。
3. 回到 `AI 协同研发` 新建 Harness 卡片，绑定 `deepwiki_run_id`。
4. 确认需求、确认设计、跑 Runtime、做 UAT。

适合：

- 多仓项目
- 依赖历史系统知识
- 需要先“看懂系统”再动手的任务

### 路径 B：文档先行

1. 在 `文档门禁` 中上传 PRD、技术方案、接口契约、DDL。
2. 执行门禁与测试方案生成。
3. 如需把方案落到开发动作，创建或绑定 Harness 卡片。
4. 在 Harness 中完成开发闭环和 UAT 收口。

适合：

- 新需求正式立项
- 需要标准化技术方案和测试方案的项目

### 路径 C：本地回归/修复

1. 在 `AI 协同研发` 直接新建卡片。
2. `repo_url` 指向本地绝对路径仓库。
3. 摘要中写明修复目标和回归范围。
4. 走完整个需求确认到 UAT 流程。

适合：

- 本地仓快速验收
- Bug 修复闭环
- 小范围收口任务

## 7. 文档门禁怎么理解

文档门禁的标准链路是：

`输入契约 -> PRD 门禁 -> 技术方案门禁 -> Coverage Graph -> 测试方案生成 -> 测试方案门禁 -> 发布`

输入建议：

- 必需：PRD、技术方案
- 强烈推荐：接口契约、DDL

结果要重点看 4 类信息：

- `缺失项`：当前无法通过的必补内容
- `风险项`：不一定阻断，但会影响质量
- `无法推导项`：缺少上下文或来源
- `知识引用`：平台实际引用了哪些知识资产

当前平台的测试方案采用双轨模式：

- 标准模板版：正式发布口径
- AI 增强版：补充边界和异常场景

发布时以标准模板版为准。

## 8. Deep Wiki 怎么理解

Deep Wiki 解决的是“系统知识怎么沉淀并持续复用”。

重点能力：

- 管理项目、仓库、版本线和仓库映射
- 生成 Deep Wiki run
- 形成 snapshot
- 浏览页面内容、图谱和图示
- 作为文档门禁和 Harness 的知识来源

推荐先做这些动作：

1. 建项目
2. 绑仓库
3. 设版本线/分支映射
4. 生成 snapshot
5. 发布可复用快照

如果 Deep Wiki 没有先沉淀好，后面的文档门禁和 Harness 很容易只剩“流程通了，但知识不够准”。

## 9. 推荐演示路径

如果要对团队做产品演示，建议按下面顺序：

1. 进入 `Deep Wiki`，展示已有项目、版本线和图谱。
2. 展示如何把 Deep Wiki 快照转成文档门禁任务。
3. 进入 `文档门禁`，展示 PRD 到技术/测试方案的生成。
4. 进入 `AI 协同研发`，展示 Harness 卡片如何承接一个具体收口任务。
5. 演示 `确认需求 -> 确认设计 -> Runtime -> UAT 通过/打回`。
6. 到 `运行编排` 和 `阶段验收` 看 trace、evidence 和结果口径。

## 10. 常见问题

### 10.1 为什么页面打开了但没有数据

通常是后端服务没起全。

优先检查：

- `admin-ui`
- `control-plane`
- `ai-gateway`
- `knowledge-base`

### 10.2 为什么 Harness 卡片不能跑

常见原因：

- 没填 `repo_url`
- 没绑 `deepwiki_run_id`
- 本地路径不存在
- 绑定的 Deep Wiki run 没有可用本地快照

### 10.3 为什么 Runtime 跑了但没有改业务文件

这是当前 V1 的设计边界。默认页面触发只会记录 `change_request` 并执行测试闭环，不会自动改业务文件。

### 10.4 为什么单测显示 skipped

说明当前工作区未识别到可执行测试命令。优先检查项目里是否存在：

- `package.json` 中的 `test`
- `mvnw` / `pom.xml`
- `gradlew` / `build.gradle`

### 10.5 为什么卡片进入 exception

常见原因：

- Runtime 准备工作区失败
- 单元测试连续失败 3 次
- 仓库来源不可解析

### 10.6 为什么 Deep Wiki、文档门禁、Harness 要分开

因为它们解决的是 3 类不同问题：

- Deep Wiki：知识沉淀
- 文档门禁：方案与交付质量
- Harness：任务执行与收口

拆开是为了职责清晰，串起来才是平台价值。

如果需要向团队介绍这套平台，建议用下面这句话：

“这是一套把 AI 工程知识沉淀、方案生成、任务流转、自动验证和阶段验收串成闭环的工程化管理平台。Deep Wiki 负责沉淀知识，文档门禁负责形成方案，Harness 负责把单个任务真正收口。”
